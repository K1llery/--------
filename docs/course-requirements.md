# Course Requirements

来源：教师 PPT《数据结构课程设计》-2026-扩展-讲义.pptx。本文档只记录对本项目有约束力的要求，便于开发、测试和验收对照。

## 交付要求

| 类别 | 要求 |
|---|---|
| 系统主题 | 基于智能体的个性化旅游系统。 |
| 开发方式 | 建议使用智能体完成系统设计、开发、测试、组内协作、文档、PPT、宣传材料等任务。 |
| 系统形态 | 最终提供真实可用系统；架构和开发语言不限。 |
| 文档材料 | 需要包含任务描述、功能需求分析、总体方案、智能体开发环境与智能体说明、数据结构和数据字典、模块设计、算法思想、测试情况、评价改进、用户使用说明。 |
| 工程过程 | 按需求分析、概要设计、详细设计、编码、测试积累文档。 |
| 版本协作 | 定义接口和开发规范，使用 GitHub 做版本管理。 |
| 成绩侧重 | 系统设计开发和验收占主要比例，报告和文档材料也计入成绩。 |

## 数据要求

| 数据项 | PPT 要求 | 本项目当前落点 |
|---|---|---|
| 景区和校园 | 数量至少 200 个，景区和校园内部可以一致。 | `datasets/prod/destinations.json` 当前 3701 个目的地。 |
| 景点 / 建筑物 / 景物 | 景区和校园内建筑物不少于 20 个，包括景点、教学楼、办公楼、宿舍楼等。 | `datasets/prod/buildings.json` 当前 40 个楼宇/建筑，`scenes.json` 和 `indoors.json` 描述场景与室内结构。 |
| 服务设施 | 类型不少于 10 种，数量不少于 50 个，如商店、饭店、洗手间、图书馆、食堂、超市、咖啡馆等。 | `datasets/prod/facilities.json` 当前 52 个设施，`facility_type` 当前 15 类。 |
| 道路图 | 建立景区和校园内部道路图，包含建筑物、服务设施等信息。 | `datasets/prod/edges.json` + `scenes.json` 被 `GraphBuilder` 抽象为有向图。 |
| 边数 | 不少于 200 条，尽量接近真实景区和校园，建议爬取真实地图数据。 | `datasets/prod/edges.json` 当前 548 条边，原始数据位于 `datasets/raw/`。 |
| 用户 | 系统用户数不少于 10 人。 | `datasets/prod/users.json` 当前 14 个用户。 |
| 持久化 | 支持数据持久化、初始数据批量导入、运行时数据交互和测试数据。 | SQLite 为默认运行存储；`datasets/prod/` 为初始化快照；测试使用临时数据隔离。 |

## 功能要求

| 功能 | PPT 要求 | 本项目落点 |
|---|---|---|
| 旅游地点推荐 | 用户按喜好选择景点和学校；系统按热度、评价、个人兴趣推荐；只看前 10 时不得依赖完全排序。 | `RecommendationService` 使用 Top-K 和 Quickselect；接口为 `/api/destinations/recommend`。 |
| 景点和学校查询 | 可按名称、类别、关键字查询；多结果可按热度和评价排序。 | `SearchService` 使用 Hash、Trie、倒排索引；接口为 `/api/destinations/search`。 |
| 单目标路线 | 用户输入目标景点或场所，系统规划当前位置到目标的最优线路。 | `RoutePlanningService.plan_single`；接口为 `/api/routes/single`。 |
| 多目标路线 | 用户输入多个目标，从当前位置出发，参观完后返回当前位置。 | 小规模使用 Held-Karp，大规模使用 Nearest Neighbor + 2-opt；接口为 `/api/routes/multi`。 |
| 地图和路径展示 | 导航功能需要图形界面，包括地图展示和输出路径展示。 | `RoutePage.vue`、`RouteMap.vue` 和路线结果面板。 |
| 路线策略 | 支持最短距离、最短时间、考虑拥挤度、交通工具限制等策略。 | 图边包含 `distance`、`congestion`、速度和 `allowed_modes`；支持 `distance`、`time`、`congestion`、`scenic` 与 `walk`、`bike`、`shuttle`、`mixed`。 |
| 室内导航 | 模拟教学楼和景区建筑内部结构，支持大门到电梯、楼层间电梯、楼层内到房间。 | `IndoorNavigationService` 和 `/api/indoor/route`，支持普通和轮椅模式。 |
| 场所查询 | 在景区或学校内部，选中地点后查找附近设施，按距离排序；不能按直线距离。 | `NearbyFacilityService` 复用单源最短路，按图距离排序；接口为 `/api/facilities/nearby`。 |
| 设施类别查询 | 用户可选择类别过滤，也可输入类别名称查找附近服务设施。 | 前端设施页和 `/api/facilities/nearby` 支持类型筛选。 |
| 旅游日记管理 | 用户可用文字、图片、视频记录旅游内容；统一管理所有用户日记。 | `DiaryService` 管理日记文本、图片字段和互动数据；视频可作为后续扩展字段。 |
| 日记浏览和评分 | 浏览量作为热度；用户浏览后可评分；日记可按热度、评价、兴趣推荐。 | `/api/diaries/{id}/view`、`/api/diaries/{id}/rate`、`/api/diaries/recommend`。 |
| 日记交流和检索 | 可按目的地查日记并按热度/评分排序；按日记名称精确查询；按内容全文检索。 | `/api/diaries/search`，倒排索引用于正文检索。 |
| 日记压缩 | 对旅游日记进行无损压缩存储。 | Huffman 压缩/解压接口：`/api/diaries/compress`、`/api/diaries/decompress`。 |
| AIGC 动画 | 根据景点或学校照片、文字描述生成旅游动画。 | `/api/diaries/{id}/aigc-animation` 生成分镜；`/api/ai/images/generate` 生成封面图。 |
| 美食推荐 | 选中景点和学校后，按热度、评价、距离排序，并按菜系过滤；前 10 不依赖完全排序；支持名称、菜系、饭店/窗口名称模糊查询。 | `foods` 接口和美食页支持展示与筛选；推荐排序复用 Top-K 思路。当前 `foods.json` 为 8 条，若验收强调 Top-10 展示，建议补到 10 条以上。 |
| 多用户 | 支持多个用户使用系统，保存多个用户信息，每个用户登录显示不同内容。 | `AuthService`、sessions、收藏、日记评分和用户态前端 store。 |

## 算法要求

| 领域 | 要求 | 本项目实现 |
|---|---|---|
| 查询 | 查询不能使用 `O(n)` 线性扫描；支持多关键字和联合查询。 | Hash、Trie、倒排索引。 |
| 排序 | 按不同关键字使用不同排序策略，并考虑实际数据量。 | 热度、评分、兴趣、距离等组合评分。 |
| Top-K | 推荐和美食只需前 10 时应做部分排序，不做全量排序。 | 小顶堆 Top-K 和 Quickselect。 |
| 图 | 地图应接近实际地图，抽象为有向图；交叉口和建筑物作为顶点。 | `Graph` + `GraphBuilder`。 |
| 最短路 | 支持最短距离、最短时间和拥挤度策略。 | Dijkstra/A*，边权按策略计算。 |
| 多点路线 | 不能只依赖阶乘级遍历；需要考虑最近邻、途程改善、回溯或分支限界等思路。 | Held-Karp 用于小规模精确解；Nearest Neighbor + 2-opt 用于大规模近似。 |
| 压缩 | 日记压缩应为无损压缩。 | Huffman。 |
| 算法分析 | 核心是算法思想、算法分析和多种算法性能/效果对比。 | `algorithms.md`、测试用例和验收讲解需保持同步。 |

## 智能体与 AI 要求

- 需要说明使用哪些智能体平台、智能体负责什么功能、如何辅助设计/开发/测试/文档。
- 需要分析 AI 的设计和开发能力、使用 AI 完成系统开发所需的人类能力、AI 无法替代且需要人工完成的工作。
- 本项目的 agent 协作说明位于 `.github/agents/*.agent.md`，仓库协作约束位于 `AGENTS.md`。
